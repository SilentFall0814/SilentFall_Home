# SilentFall_Home 性能优化与宝塔部署文档设计

## 1. 背景与目标

当前项目是一个基于 `Vite + React + TypeScript + Tailwind CSS` 的单页前端站点，首屏包含较多动画与实时交互效果，包括：

- 基于 `canvas` 的动态点阵背景
- 鼠标透视跟随遮罩与拖尾
- `framer-motion` 驱动的多层视差与滚动联动
- `Lenis` 平滑滚动

现阶段已经确认以下事实：

- 当前仓库仅包含前端代码，不包含后端接口、数据库或服务端应用
- 当前项目 `npm run build` 会失败，存在未使用符号导致的 TypeScript 构建错误
- 用户要求对当前仓库做真实落地优化
- 优化时优先保留现有首屏视觉效果，不采用大幅删减动画的激进方案
- 部署文档直接以仓库内 `Markdown` 文件形式交付

本次工作的总体目标如下：

- 在尽量保留首屏视觉辨识度的前提下，降低页面卡顿、输入延迟和高频重绘成本
- 修复现有构建问题，提升项目工程质量与可维护性
- 输出一份适用于当前静态前端项目的宝塔面板部署文档
- 对无法在当前仓库真实落地的服务端项，明确标注为“宝塔部署建议配置”，不伪装为已完成代码能力

## 3. 现状问题判断

基于当前代码结构，主要性能风险如下：

### 3.1 工程结构风险

- `App.tsx` 承担了数据定义、交互逻辑、动画逻辑、绘制逻辑和页面布局，文件过大，难以维护
- 首屏相关逻辑高度耦合，后续定位瓶颈和做定向优化成本较高

### 3.2 渲染性能风险

- 鼠标位移和滚动进度通过 React 状态持续驱动，可能引发高频组件更新
- `DotGridCanvas` 在每帧中对大量点位进行遍历和绘制，且范围覆盖较大
- 透视遮罩拖尾动画持续运行时，对移动设备和低性能设备压力较大
- 多处 `framer-motion` 运动值叠加，增加首屏首次交互后的运行成本

### 3.3 资源与构建风险

- 当前构建配置较基础，缺少更细粒度的产物控制
- 自定义字体为本地 `ttf`，需要确认是否存在资源体积偏大问题
- 站点图标、静态资源、缓存策略尚未形成面向生产部署的规范

### 3.4 交互体验风险

- 页面滚动体验依赖 `Lenis` 与动画联动，一旦帧率下降，体感卡顿会较明显
- 首屏动画较多，若没有按设备能力进行限制，移动端与低性能端流畅度容易下降

## 4. 设计原则

本次设计遵循以下原则：

- 保留首屏核心观感，不做肉眼明显的风格阉割
- 优先降低实现成本，而不是先牺牲视觉表现
- 高频数据尽量留在 `ref`、`motion value`、`requestAnimationFrame` 中，避免频繁触发 React 渲染
- 仅在真正需要时启用昂贵动画，非首屏或低性能场景主动降级
- 工程重构必须服务于本次性能目标，不做无关重构
- 所有结论以实际构建和实测结果为准，不虚报指标

## 5. 方案对比与决策

### 5.1 方案一：激进提分方案

做法：

- 大幅删除首屏动效
- 去掉 `Lenis`
- 去掉复杂拖尾遮罩
- 缩减 `canvas` 背景计算

优点：

- 得分提升最快
- 风险较低

缺点：

- 破坏当前项目视觉识别度
- 不符合“优先保留现有首屏视觉效果”的要求

### 5.2 方案二：平衡保真方案

做法：

- 保留点阵背景、透视跟随、拖尾遮罩、滚动层次感
- 通过实现方式优化来降低运行成本
- 拆分组件与 Hook，提高局部优化能力
- 增加按设备能力降级与按区域激活策略

优点：

- 首屏观感保留度高
- 性能与维护性可以同时提升
- 最适合当前项目

缺点：

- 实现复杂度高于简单删减动画

### 5.3 方案三：最小改动方案

做法：

- 仅修复构建错误
- 做少量 CSS 和构建配置优化

优点：

- 改动小
- 上线风险低

缺点：

- 无法满足“全面优化”的目标
- 难以形成有说服力的优化结果

### 5.4 最终决策

采用方案二，也就是“平衡保真方案”。

原因如下：

- 当前项目的核心价值之一正是首屏视觉表现
- 现有问题更像是实现成本偏高，而不是设计本身不可保留
- 通过组件拆分、状态下沉、动画激活控制和绘制限流，有较大概率在不明显牺牲观感的前提下提升流畅度

## 6. 前端优化设计

### 6.1 结构拆分设计

当前 `App.tsx` 将拆分为更清晰的职责边界，建议结构如下：

- `src/components/layout/Navbar.tsx`
- `src/components/hero/PerspectiveHeroSection.tsx`
- `src/components/hero/DotGridCanvas.tsx`
- `src/components/hero/SignatureLetters.tsx`
- `src/components/sections/AboutSection.tsx`
- `src/components/sections/OtherPagesSection.tsx`
- `src/components/sections/ContactSection.tsx`
- `src/hooks/useViewportProfile.ts`
- `src/hooks/useParallaxPointer.ts`
- `src/hooks/usePerspectiveTrail.ts`
- `src/hooks/useSmoothScroll.ts`
- `src/utils/performance.ts`

目标：

- 将布局结构、交互状态、绘制逻辑、运动逻辑解耦
- 降低单文件复杂度
- 便于对高成本区域独立分析和替换

### 6.2 高频状态优化设计

重点优化以下问题：

- 鼠标和滚动数据尽量不再通过 React `setState` 高频同步
- 对仅用于绘制和位移计算的数据，优先改为 `ref` 或运动值直接消费
- 对需要展示反馈的少量 UI 状态，例如复制邮箱提示，继续保留 React 状态

具体策略：

- `pointer` 与 `scrollProgress` 从“频繁触发组件重渲染”的模式，调整为“动画层内部读取最新值”的模式
- 在 `canvas` 绘制和拖尾遮罩中，优先使用 `ref` 存储实时值
- 将需要每帧更新的逻辑集中在 `requestAnimationFrame` 回路中，避免多个独立循环相互竞争

### 6.3 Hero 区域动画优化设计

#### 点阵背景优化

- 保留点阵背景的视觉效果
- 控制绘制范围与点密度，避免过度遍历
- 根据设备能力动态调整 `spacing`、透明度桶精度和绘制区域高度
- 对低性能场景减少点数量，但不直接消失
- 尽量复用预计算结果，减少每帧重复计算项

#### 鼠标透视与拖尾优化

- 保留圆形与椭圆遮罩切换的核心表现
- 在触控设备、窄屏设备、低性能设备上启用更轻量的模式
- 将拖尾计算保留在单个动画循环中
- 当鼠标离开 Hero 且拖尾收敛后，及时停止动画循环

#### 滚动与视差优化

- 保留首屏滚动层次感
- 缩减多层位移幅度，避免“视觉差异很小但计算成本不低”的运动链
- 仅在 Hero 可见区内启用复杂联动，滚出首屏后关闭或冻结高成本效果

### 6.4 平滑滚动优化设计

`Lenis` 不会直接删除，但会做保守处理：

- 保留其在桌面端的顺滑体验
- 在窄屏、触控设备、减少动态偏好下主动降级
- 仅保留必要参数，避免过度平滑带来的拖滞感
- 确保锚点导航逻辑与滚动逻辑一致，避免双重滚动实现冲突

### 6.5 资源优化设计

将对以下资源进行处理：

- 修复未使用符号导致的构建失败
- 检查字体资源使用方式，必要时补充字体回退策略与说明
- 清理未使用代码与冗余工具函数
- 优化站点图标与元信息引用方式
- 根据当前项目资源情况，补充适合静态部署的缓存说明

如项目内确有大图片资源，将进一步处理为更优格式；若当前仓库几乎无图片资源，则在文档中说明建议但不伪造处理结果。

### 6.6 构建配置优化设计

会在 `Vite` 构建层面进行适度优化，包括：

- 明确生产构建输出行为
- 必要时补充手动分包策略
- 降低无意义的大块产物风险
- 为宝塔静态部署补充缓存与压缩建议

由于当前项目是静态前端，不会为了追求复杂度而强行引入额外构建插件；优先采用当前技术栈内可维护的方案。

## 8. 错误处理与风险控制

### 8.1 已知风险

- 动画优化过程中可能导致视觉细节发生轻微变化
- 组件拆分过程中可能引入交互回归
- 不同设备上对 `backdrop-filter`、`clip-path`、`canvas` 的表现可能存在差异

### 8.2 控制策略

- 每次重构后都保持可构建
- 以首屏为核心做逐步替换，避免一次性重写全部动画
- 对移动端和低性能端优先提供稳定体验
- 保留功能一致性，避免链接、锚点、复制按钮等基础功能回归

## 9. 测试策略

本次以高价值验证为主，不添加低价值测试噪音。

主要验证方式：

- 构建验证：`npm run build`
- 诊断检查：修改后文件的编辑器诊断结果
- 本地交互验证：导航跳转、首屏滚动、外链跳转、邮箱复制
- 多视口检查：桌面端、移动端
- 性能验证：Lighthouse 与构建产物对比

如果在拆分过程中出现纯函数或工具函数，且针对性测试能显著降低回归风险，可补充少量聚焦测试；否则不强行增加测试框架和样板代码。
