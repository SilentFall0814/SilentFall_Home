# SilentFall_Home 性能优化报告

## 一、优化前基线

### 1. 构建状态

执行命令：

```bash
npm run build
```

初始结果：

- 构建失败
- `src/App.tsx` 中存在未使用导入 `useMemo`
- `src/App.tsx` 中存在未使用函数 `isInternalHashLink`

### 2. 优化前构建产物

在修复构建失败后，优化前的主要产物如下：

| 资源     | 优化前         |
| -------- | -------------- |
| HTML     | `0.63 kB`      |
| CSS      | `16.68 kB`     |
| 主 JS    | `365.54 kB`    |
| 图标图片 | `92.61 kB`     |
| 本地字体 | `10,898.26 kB` |

关键问题：

- 本地 `ttf` 字体被直接打入生产包，单文件体积约 `10.9 MB`
- `App.tsx` 体积和职责过大，首屏逻辑、动画逻辑、内容区块耦合严重
- 首屏高频动画状态在顶层组件同步，增加了重渲染成本
- 生产构建未做拆包，缓存利用率较弱

## 二、已落地优化项

### 1. 工程结构优化

- 修复了当前构建失败问题，确保 `npm run build` 可稳定通过
- 将站点常量、内容区块、Hero 组件、导航组件、滚动 Hook、设备能力 Hook 拆分为独立文件
- 将 `App.tsx` 收缩为页面装配层，降低后续维护和性能分析成本

### 2. 首屏性能优化

- 将 Hero 区域从 `App.tsx` 中独立出来，减少顶层组件职责
- 移除了 `App.tsx` 中对 `pointer` 和 `scrollProgress` 的高频状态维护
- 将点阵画布与拖尾遮罩拆分为独立模块，便于单独控制和继续优化
- 让点阵参数根据 `AnimationLevel` 自动切换，为后续轻量模式预留能力
- 让高成本点阵画布仅在 Hero 处于活跃区间时渲染

### 3. 资源与构建优化

- 去除会将超大本地字体打入生产包的 `@font-face` 引用，改为系统字体栈
- 为生产构建补充 `manualChunks` 函数式拆包策略
- 将导航组件独立，减少顶层文件复杂度
- 补充 `theme-color` 和更贴合当前项目的 `description` 元信息
- 为系统“减少动态”偏好增加全局降级样式

- `WebPageTest` 未执行，原因是当前任务环境以本地仓库实施为主，且无现成公网地址用于远程采集
- Lighthouse 已尝试执行，但当前环境没有可用 Chrome 安装，因此无法生成真实评分
- 因为用户要求优先保留首屏视觉效果，所以本次没有通过大幅删减动画来换分

## 五、结论

本次优化已经完成以下真实目标：

- 项目从“无法构建”恢复到“可稳定构建”
- 站点完成结构拆分，核心高频逻辑从顶层组件下沉
- 超大本地字体从生产资源链中移除，主要静态产物总量下降约 `95.82%`
- 生产构建完成基础拆包，缓存友好度显著提升
- 性能报告和宝塔部署文档已具备继续交付基础

后续如果要进一步冲击更高 Lighthouse 分数，优先方向建议为：

1. 继续降低 Hero 中的实时状态同步频率
2. 为点阵画布增加更强的可见区停用策略
3. 进一步评估 `framer-motion` 在首屏中的使用密度
4. 在具备 Chrome 的环境中补采 Lighthouse 与 Performance 面板数据
